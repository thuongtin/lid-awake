## Findings Considered And Rejected

Second audit (2026-06-28):

- No findings were rejected in this pass. The user asked to plan all vetted

Third audit (2026-07-02, commit `5d89d38`) - rejected or deferred so nobody
re-audits them:

- XPC completion gate shared across calls (claimed leak in
  `ClosedLidHelperService`): refuted; the gate is created per
  `withRemoteObject()` call, not per service instance.
- `evaluate()` re-entrancy racing in-flight XPC: refuted; guarded by
  `isChangingClosedLidMode` plus the `closedLidModeChangeID` latch.
- Sparkle downgrade protection in-app: rejected; Sparkle's default version
  comparison only offers newer versions, and the appcast travels over HTTPS
  with EdDSA-signed archives. Custom delegate logic not worth the risk.
- Helper-binary integrity re-verification at launch: rejected as by-design;
  bundle compromise requires defeating Developer ID signing/notarization,
  documented in `docs/power-model.md` trust boundary.
- Helper NSLog of rejected client PID: rejected; PIDs are not sensitive.
- `SettingsStore.save` silently ignoring encode failures and missing settings
  schema versioning: deferred; `UserSettings` decode already tolerates missing
  keys per-field, and encoding a plain Codable struct cannot realistically
  fail today. Revisit only when the settings schema next changes shape.
- `check.sh` running `swift build` after `swift test`: rejected;
  SwiftPM incremental builds make the second invocation nearly free.
- AppModel god-file split (1014 lines): deferred; real but L-effort,
  MED-risk refactor with thin payoff while the file is still navigable.
  Reconsider if closed-lid logic grows again.
- Typed errors instead of `String?` fields + message-substring matching
  (`isClosedLidReadinessError`): deferred as backlog; fragile pattern but
  localized and test-covered today.
- `script/` vs `scripts/` directory split and packaging-script duplication
  (signing-identity resolution repeated in 3 scripts): deferred as backlog
  polish; consolidate when a packaging script next changes anyway.
- StatusItemController unit tests via mocked NSStatusBar: rejected;
  AppKit mocking cost exceeds the risk in 150 lines of glue.
- Universal (Intel) binary support: not planned; documented limitation and a
  product decision for the maintainer, not an audit fix.
- Agent-session machinery (`AgentSession`, `AgentHookEvent`, `AgentKind`,
  `AgentSessionMerger`, `AppModel.sessions`): NOT planned in this batch by
  deliberate choice. `docs/power-model.md` states the app does not detect
  agents, yet ~260 lines of tested agent-domain code remain reachable only
  through the synthetic "Manual Hold" session. Two mutually exclusive paths,
  maintainer must pick: (a) remove the machinery and simplify
  `WakePolicyCoordinator` input (tech-debt cleanup, M), or (b) revive the
  original AgentAwake direction with a hook ingestion entry point and
  sessions UI (direction, M-L, see plan 001 for the original vision). Do not
  execute either without an explicit maintainer decision.
